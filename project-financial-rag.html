What This Project Is

A production-style Retrieval-Augmented Generation (RAG) platform that answers natural-language questions about public companies — Apple, Microsoft, Amazon, NVIDIA, and Realty Income (a REIT) — using their actual SEC filings and recent financial news. Instead of returning a list of links like a search engine, it reads the underlying documents and generates a direct, cited answer, with every claim traceable back to the specific filing or article it came from.

It's built as a full-lifecycle system: real data ingestion, a Medallion-architecture data pipeline, evaluation with measured accuracy, MLOps observability, Infrastructure-as-Code, CI/CD, and a public, live-deployed application — not a notebook demo.

The Problem

Public companies are legally required to disclose detailed financial and risk information — but that information is buried in dense, jargon-heavy documents. A single 10-K annual report routinely runs over 100 pages. Finding the answer to a specific question — "What tariff risks does this company disclose?" — means manually reading or keyword-searching through the entire filing.

This isn't a data scarcity problem — the information is public and abundant. It's a data accessibility problem. Institutional tools like Bloomberg Terminal have solved this for large firms willing to pay tens of thousands of dollars a year. For individual investors, students, or smaller firms, that gap has largely remained open — until foundation models made a lighter-weight version of the same capability newly practical.

The Solution

A RAG system that treats the LLM as one component in a larger, engineered pipeline — not the whole application. Raw filings and news are ingested, cleaned, chunked, and embedded into a searchable vector space. When a user asks a question, the system retrieves the most relevant pieces of real text and generates an answer using only that retrieved content — explicitly instructed to cite its source for every claim, and to say so plainly when the data doesn't support an answer, rather than guessing.

Real Case Scenario

// question

"What are the key risk factors in Apple's latest 10-K?"

What happens under the hood: the system embeds the question into the same vector space as the document corpus, retrieves the 5 most semantically relevant chunks from Apple's actual filings, and passes them to the LLM with strict instructions to answer only from those sources.

Result: a structured answer covering intellectual property risk, third-party developer dependence, product liability exposure, and macroeconomic/tariff risk — each sentence citing the specific source chunk it came from, so the answer can be independently verified against the real filing.

// question — the trust-building counter-example

"Summarize analyst sentiment on REITs this quarter"

Why it can't answer: this is a question the underlying data genuinely doesn't answer, since the corpus contains financial disclosures, not analyst opinions.

Result: the system correctly responds that it cannot answer, rather than fabricating a plausible-sounding guess. This was a deliberate test case, and it's the single most important behavior in a system meant to be trusted with financial research.

Key Design Decisions

01

Medallion architecture over a flat pipeline

Separating Bronze/Silver/Gold made a real, non-obvious data-quality bug isolatable to a single layer, rather than tangled across the whole system.

02

Local embeddings in dev, API-based in production

Re-architected the deployed version to use OpenAI's embedding API after diagnosing that the local model's baseline memory footprint — not dataset size — exceeded the hosting platform's resource limits.

03

Strict, tested citation requirements

Iterating from vague, blanket citation toward requiring a citation on every individual factual sentence, verified through a formal evaluation harness, meaningfully improved answer trustworthiness.

04

Deliberately testing for refusal, not just correctness

The evaluation set includes a question the data cannot answer, specifically to verify the system fails safely (declines) rather than hallucinating.

The Foundation of the Pipeline

  • Data sources: SEC EDGAR (public filings API) and NewsAPI — both free, public, real-time
  • Storage: Delta Lake on Azure Databricks, providing ACID transactions and schema enforcement across the pipeline
  • Retrieval: Vector similarity search over chunked, embedded text
  • Generation: OpenAI's language models, constrained by a citation-strict prompt
  • Observability: MLflow experiment tracking on every query — question, sources retrieved, latency, and output, all logged
  • Infrastructure: Terraform-managed Azure resources (storage, secrets management via Key Vault), CI/CD via GitHub Actions running automated tests on every commit

Benefits of This Approach

  • Speed of research — a question that would take 10-20 minutes of manual reading through a filing is answered in seconds, with the source still fully verifiable.
  • Trustworthiness through citation — every claim is traceable to its exact source, and the system is explicitly designed to decline rather than fabricate when the answer isn't supported by the data. This directly addresses the core risk of using generative AI for factual research: unverifiable, confidently-stated errors.
  • Accessibility — this pattern makes institutional-grade document research capability available without an institutional-grade budget, narrowing a real gap between individual investors/researchers and well-resourced firms.

Scaling Scenario: What Changes at 1,200 Tickers

The current system covers 5 companies by design — enough to prove the architecture works on real, messy data without the cost and complexity of full market coverage. But the architecture itself doesn't change with scale: ticker coverage is defined by a configuration dictionary, not hardcoded logic, so expanding coverage is a data-volume change, not a re-engineering effort.

What a 1,200-ticker version would actually enable: portfolio-wide research instead of single-company lookups.

An analyst covering an entire sector — or an individual investor screening across the S&P 1200 — could ask comparative questions ("which companies in my portfolio disclosed new supply chain risk this quarter?") instead of checking each filing individually. At that scale, the tool shifts from "read this one document faster" to "monitor an entire universe of companies for material changes automatically" — closer to what institutional research platforms actually provide, but built on the same core pipeline demonstrated here.

What would need to change operationally, not architecturally: ingestion would move from a manual/scheduled batch job to a properly rate-limited, incremental pipeline (SEC and NewsAPI both cap request rates); the vector store would need a managed, horizontally-scalable backend instead of a local instance; and cost monitoring would become a first-class concern rather than an afterthought, since 1,200 tickers means meaningfully more embedding and generation volume — though even at that scale, embedding costs alone remain under $10 total, based on measured per-ticker costs from the current pipeline.
