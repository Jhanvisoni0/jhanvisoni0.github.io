Retail Banking Customer Analytics Platform

Source Code ↗

An end-to-end Azure data engineering platform that unifies fragmented banking data across 3 source systems into a single, automated analytics pipeline — enabling real-time risk monitoring and executive reporting for a retail bank.

The Problem

FirsTrust Bank managed customer profiles, loan portfolios, and transaction records across 3 completely separate, disconnected systems with no unified view. The risk team spent 4 days every week manually reconciling these systems in Excel before any meaningful analysis could happen. There was no automated alerting for high-risk accounts, no single customer view across products, and no way to detect anomalous transactions without manual inspection of raw exports.

The Solution

A fully automated Medallion Architecture pipeline on Azure that ingests raw data from all 3 source systems daily, transforms it through Bronze → Silver → Gold layers using PySpark, computes banking-specific KPIs and risk scores via an MLflow-tracked ML model, and serves a live 3-page Power BI executive dashboard via DirectQuery on Databricks SQL — reducing the reporting cycle from 4 days to 15 minutes.

How It Works — Pipeline Flow

Ingest

STEP 01

Azure Data Factory pulls 3 CSV exports (customers, loans, transactions) from the source system into ADLS Gen2 Bronze on a daily scheduled trigger.

Store Raw

STEP 02

Bronze Delta tables preserve data exactly as received — no transformations, full audit trail.

Clean

STEP 03

Silver layer enforces schemas, deduplicates records, standardizes delinquency status codes, and extracts time features from timestamps.

Compute KPIs

STEP 04

Gold layer joins all 3 sources into a Customer 360 view: loan-to-deposit ratio, 30/60/90-day delinquency flags, rolling transaction velocity, account dormancy score.

Score Risk

STEP 05

Isolation Forest model trained on engineered features flags anomalous transactions; results written back to Gold, full experiment tracked in MLflow.

Orchestrate

STEP 06

Databricks Workflows runs the full Bronze → Silver → Gold → Risk pipeline on a daily 6AM schedule with retry logic and email failure alerts.

Serve

RESULT

Power BI connects via DirectQuery to Databricks SQL Warehouse; the dashboard always shows live data, no manual refresh needed.

Real Case Scenario

// Monday morning, FirsTrust Bank risk desk

1,500 transactions flagged HIGH_RISK overnight

A risk analyst arrives Monday morning. Instead of opening 3 separate system exports and spending half the day in Excel, they open the Power BI Risk Flags page. They see 1,500 transactions flagged HIGH_RISK overnight, sorted by anomaly score. They click a flagged account — a customer whose transaction amount was 4 standard deviations above their own historical average at 2AM on a Saturday — and escalate it to the fraud team before the business day starts.

The entire process takes 10 minutes. The pipeline that made it possible ran automatically at 6AM and processed 50,000 transactions without anyone touching a keyboard.

Key Design Decisions

01

Medallion Architecture over transform-on-ingest

Raw data is preserved in Bronze exactly as the source system produced it. If transformation logic changes, you re-process from Bronze without ever going back to the source system.

02

Delta Lake over Parquet

Delta's MERGE operation handles upserts safely as loan statuses update and transactions get corrected. Time travel gives compliance point-in-time reconstruction without a separate archiving system.

03

Batch scoring with MLflow over a live endpoint

Risk scores computed once daily is appropriate for a daily reporting cycle and far cheaper than a live serving endpoint. MLflow tracks every run — an auditable model history required in financial services.

04

DirectQuery over Import mode

A risk dashboard must show today's flagged accounts, not yesterday's cached snapshot. DirectQuery queries Databricks SQL live on every interaction.

05

ADF for ingestion, not a custom script

Azure Data Factory provides scheduling, monitoring, retry logic, and connectivity to 90+ sources out of the box — a visual, auditable pipeline instead of a script on someone's laptop.

The Foundation of the Pipeline

  • Idempotency — every notebook can be re-run safely without creating duplicates or corrupting data. Delta MERGE handles upserts; checkpoints track ingestion state. A failed run can be retried with no manual cleanup.
  • Separation of concerns — data movement (ADF), transformation (Databricks), orchestration (Workflows), and presentation (Power BI) are fully decoupled. Each layer can be updated independently without touching the others.
  • Observability — every layer produces a verifiable output — ADF shows rows written per run, Bronze/Silver row counts validate ingestion, data quality checks compare Bronze vs. Silver, MLflow logs every model metric, Workflow runs show per-task status, and Delta history provides a full audit trail from raw file to Gold table.

Benefits

  • Reporting cycle: 4 days → 15 minutes — what previously required manual reconciliation across 3 systems in Excel now runs automatically every morning, with data ready before the working day starts.
  • Proactive risk detection instead of reactive reporting — the Isolation Forest model flags anomalous transactions nightly — accounts are reviewed before customers call to report fraud, not after. 1,500 transactions were flagged in the first run, representing patterns invisible to rule-based systems.
  • Full audit trail from raw file to dashboard number — every number on the Power BI dashboard can be traced back through Gold → Silver → Bronze → original source file. Delta Lake's version history means you can query the exact state of any table on any past date — critical for regulatory compliance in banking.

Big Data Scenario

At portfolio scale this pipeline processes 50,000 transactions. Here's how the same architecture scales to a real bank's volume.

A mid-sized retail bank processes roughly 2 million transactions per day across 500,000 active customers.

The architectural decisions made at portfolio scale — idempotency, partition strategy, separation of concerns — are the same ones that make the pipeline viable at 2 million transactions per day. The scale changes the implementation details, not the design principles.

At that scale:

  • Bronze ingestion switches from batch CSV reads to Auto Loader with trigger(availableNow=True) — incrementally processing only new files each run rather than reloading the full dataset, reducing ingestion time from hours to minutes
  • Silver partitioning by transaction year and date (not just year) means Spark reads only the relevant day's partition rather than scanning 730 million rows for a single day's analysis
  • Gold window functions for rolling velocity use rangeBetween on a timestamp column — Spark's query planner pushes the partition filter down before computing the window, avoiding a full shuffle of 2 billion rows
  • Isolation Forest moves from a single Pandas .fit() call to a distributed implementation using Spark MLlib or a pandas_udf that partitions scoring across the cluster — eliminating the Pandas memory ceiling
  • The Workflow adds a data quality gate between Bronze and Silver — if row counts drop more than 5% from the previous day, the pipeline stops and pages the on-call engineer rather than silently propagating bad data to Gold
  • The SQL Warehouse scales from 2X-Small to Medium during business hours (when Power BI queries are heaviest) and back to 2X-Small overnight — auto-scaling keeps costs proportional to actual query load
